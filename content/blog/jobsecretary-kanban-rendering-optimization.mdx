JobSecretary의 칸반 보드에서 카드를 드래그할 때, 이동 중인 카드뿐 아니라 여러 카드가 함께 다시 렌더링되는 현상을 발견했습니다. React Developer Tools Profiler로 같은 드래그 동작을 측정한 뒤 드래그 상태를 처리하는 영역과 카드 정보를 표시하는 영역을 분리했습니다. 같은 시나리오에서 전체 커밋은 **271회 → 98회** (**63.8% 감소**), 비활성 컴포넌트 리렌더링은 **268회 → 0회** (**100% 감소**)로 줄었습니다.

같은 동작을 개선 전과 후에 각각 기록하고, React Developer Tools Profiler에서 렌더링 변화를 비교했습니다. 아래에서는 JobSecretary의 컴포넌트 구조에서 문제가 발생한 지점과 변경 과정을 정리합니다.

## 문제 상황

React Developer Tools의 Profiler에서 동일한 칸반 목록의 카드를 선택해 다른 위치에 놓는 동작을 측정했습니다. 드래그 중 위치가 바뀌는 것은 예상할 수 있는 동작이었지만, 이동하지 않은 카드까지 같은 렌더링 흐름에 포함되고 있었습니다.

측정 결과 한 번의 드래그 과정에서 총 **271회**의 커밋이 발생했고, 실제 변경이 없는 비활성 컴포넌트 렌더링도 **268회** 발생했습니다.

> 측정 시나리오: React Developer Tools Profiler에서 동일한 칸반 카드 목록의 카드를 드래그해 다른 위치에 놓는 과정

이 수치는 브라우저와 실행 모드, 카드 수에 따라 달라질 수 있습니다. 따라서 절대적인 라이브러리 성능 수치가 아니라, 같은 프로젝트에서 구조 변경 전후를 비교하기 위한 기준으로 사용했습니다.

개선 전·후 동작은 다음 영상으로 비교할 수 있습니다. 영상은 동일한 프로젝트에서 측정 전후의 흐름을 기록한 자료이며, 영상만으로 렌더링 횟수를 판단하지 않고 Profiler 측정 결과와 함께 해석했습니다.

## 원인: 드래그 상태와 카드 콘텐츠가 같은 렌더링 범위에 있었다

문제 발생 당시에는 `dnd-kit`의 `SortableContext`와 `useSortable`을 사용했습니다. `SortableContext`는 정렬 가능한 아이템 목록과 정렬에 필요한 정보를 하위 컴포넌트에 제공하고, `useSortable`은 이 정보를 읽어 드래그에 필요한 `transform`, `transition`, `setNodeRef`, 이벤트 핸들러 등을 반환합니다. [dnd-kit Sortable 공식 문서](https://dndkit.com/legacy/presets/sortable/overview/)

구조를 단순화하면 다음과 같습니다.

```text
SortableContext
 ├─ KanbanCard A: useSortable()
 ├─ KanbanCard B: useSortable()
 └─ KanbanCard C: useSortable()
```

카드를 드래그하는 동안에는 활성 카드와 위치, 충돌 대상, transform 같은 값이 계속 변합니다. `useSortable`을 사용하는 카드가 `SortableContext`의 상태를 직접 읽고 있으면 Context 변화가 해당 Hook을 사용하는 카드 컴포넌트의 렌더링에 영향을 줄 수 있습니다.

당시에는 드래그 상태를 처리하는 코드와 회사명·직무·마감일을 표시하는 코드가 같은 카드 컴포넌트 안에 있었습니다. 결과적으로 카드 위치를 계산하기 위한 상태 변화가 카드 콘텐츠를 표시하는 영역까지 같은 렌더링 범위에 포함되었습니다.

`React.memo`도 적용했지만 이 문제를 충분히 줄이지 못했습니다. `React.memo`는 부모가 전달하는 Props를 비교해 값이 같으면 렌더링을 건너뛸 수 있게 하는 최적화입니다. 반면 컴포넌트가 직접 사용하는 Context의 값이 변경되면 Props가 같아도 다시 렌더링될 수 있습니다. [React.memo 공식 문서](https://react.dev/reference/react/memo)

따라서 이번 문제의 핵심은 메모이제이션을 적용했는지가 아니었습니다. 드래그 상태를 구독하는 컴포넌트와 카드 콘텐츠를 표시하는 컴포넌트가 같은 경계에 있었던 것이 원인이었습니다.

## 해결 과정 1. 드래그 상태 전달 방식 변경

칸반 구현을 `@hello-pangea/dnd` 기반으로 변경했습니다. 현재 저장소에서는 `DragDropContext`로 전체 드래그 영역을 만들고, `Droppable`과 `Draggable`로 컬럼과 카드를 구성합니다. [칸반 보드 코드](https://github.com/kimjunha1231/JobSecretary/blob/main/features/document-kanban/ui/kanban-board.tsx), [칸반 컬럼 코드](https://github.com/kimjunha1231/JobSecretary/blob/main/features/document-kanban/ui/kanban-column.tsx)

`Draggable`은 자식 함수에 드래그 연결 정보와 현재 드래그 상태를 전달합니다.

```tsx
<Draggable draggableId={id} index={index}>
  {(provided, snapshot) => (
    <KanbanCard
      provided={provided}
      isDragging={snapshot.isDragging}
    />
  )}
</Draggable>
```

이 함수는 애플리케이션에서 임의로 실행하는 것이 아니라 라이브러리가 실행합니다. `provided`에는 카드의 DOM을 드래그 기능과 연결하기 위한 `innerRef`, `draggableProps`, `dragHandleProps`가 포함됩니다. `snapshot.isDragging`은 현재 카드가 드래그 중인지 알려주므로 드래그 중 스타일을 적용할 때 사용할 수 있습니다.

이 방식은 Render Props 패턴에 해당합니다. Render Props는 성능을 자동으로 개선하는 기능이 아니라, 라이브러리가 계산한 값을 함수의 인자로 전달하고 필요한 컴포넌트가 사용하도록 하는 API 패턴입니다. [React Render Props 문서](https://legacy.reactjs.org/docs/render-props.html)

## 해결 과정 2. 카드 기능과 콘텐츠 분리

`KanbanCard`는 드래그 기능에 필요한 DOM 연결과 스타일을 담당하고, `KanbanCardContent`는 실제 카드 정보를 표시하도록 역할을 나눴습니다.

```text
Draggable
 └─ KanbanCard
     ├─ provided.innerRef
     ├─ draggableProps
     ├─ dragHandleProps
     └─ KanbanCardContent
         └─ application, onDelete
```

`KanbanCardContent`에서는 드래그 상태를 직접 구독하지 않고 카드 데이터와 삭제 함수만 Props로 받도록 구성했습니다. 이후 콘텐츠 컴포넌트에 `memo`를 적용했습니다. [KanbanCard 구현](https://github.com/kimjunha1231/JobSecretary/blob/main/features/document-kanban/ui/kanban-card.tsx)

이제 드래그 중인 카드의 위치나 스타일이 변경되어 `KanbanCard`가 다시 렌더링되더라도, `KanbanCardContent`에 전달되는 Props가 같으면 콘텐츠 렌더링을 건너뛸 수 있습니다.

핵심은 Render Props가 모든 리렌더링을 막았다는 데 있지 않습니다. 드래그 상태가 바뀌는 컴포넌트와 카드 정보를 표시하는 컴포넌트를 분리해, `memo`가 적용될 수 있는 렌더링 경계를 만든 데 있습니다.

## 해결 과정 3. 드래그 중인 카드의 DOM 위치 분리

드래그 중인 카드는 `React Portal`을 통해 `document.body` 아래에 표시했습니다.

```tsx
if (isDragging && typeof document !== "undefined") {
  return createPortal(card, document.body);
}
```

Portal의 목적은 렌더링 성능 개선이 아니었습니다. React 트리와 이벤트 흐름은 유지하면서 실제 DOM 위치만 바꿔, 스크롤 컨테이너의 `overflow`, `z-index`, 레이아웃 간섭을 줄이기 위해 사용했습니다. [React createPortal 공식 문서](https://react.dev/reference/react-dom/createPortal)

현재 구현에도 윈도우 단위 스크롤 환경에서 드래그 중인 카드를 `document.body`로 옮기는 방식이 자동 스크롤을 처리하는 데 안정적이라는 설명이 남아 있습니다. [KanbanCard의 Portal 적용 코드](https://github.com/kimjunha1231/JobSecretary/blob/main/features/document-kanban/ui/kanban-card.tsx)

## 측정 결과

같은 드래그 시나리오를 React Developer Tools Profiler에서 구조 변경 전후로 다시 측정했습니다.

| 항목 | 변경 전 | 변경 후 | 개선 효과 |
| --- | ---: | ---: |
| 전체 커밋 횟수 | 271회 | 98회 | **63.8% 감소** |
| 비활성 컴포넌트 렌더링 | 268회 | 0회 | **100% 감소** |

전체 커밋은 **271회 → 98회** (**63.8% 감소**)로 감소했고, 비활성 컴포넌트 렌더링은 **268회 → 0회** (**100% 감소**)로 줄었습니다.

이 결과를 라이브러리 교체 하나의 효과로 해석하지 않았습니다. 드래그 상태 전달 방식 변경, `KanbanCard`와 `KanbanCardContent` 분리, 콘텐츠의 `memo` 적용, Portal을 통한 DOM 위치 분리를 함께 적용한 결과입니다.

이번 측정에서 확인한 것은 렌더링 범위의 변화입니다. 실제 사용자가 느끼는 조작감이나 모든 환경에서의 성능 개선을 주장하려면 브라우저, 실행 모드, 카드 수를 고정한 추가 측정이 필요합니다.

## 정리

이번 문제를 해결하면서 먼저 확인한 것은 어떤 라이브러리가 더 좋은지가 아니라, 드래그 상태를 어느 컴포넌트가 구독하고 있는지였습니다.

`useSortable`이 `SortableContext`의 상태를 읽는 구조에서는 드래그 중 Context 변화가 Hook을 사용하는 카드에 전달될 수 있습니다. 카드 콘텐츠까지 같은 컴포넌트에 포함되어 있으면 `React.memo`만으로는 렌더링 범위를 충분히 줄이기 어렵습니다.

변경 후에는 Render Props로 드래그 연결 정보와 상태를 필요한 컴포넌트에 전달하고, 카드 콘텐츠는 일반 Props만 받도록 분리했습니다. `memo`는 드래그 상태를 직접 구독하지 않는 콘텐츠 영역에 적용했고, Portal은 성능 최적화가 아니라 DOM 배치와 레이아웃 간섭을 줄이는 목적으로 사용했습니다.

결과적으로 이번 최적화의 핵심은 라이브러리 교체 자체보다 드래그 로직과 카드 콘텐츠의 렌더링 경계를 분리한 데 있습니다.

## 자주 묻는 질문

### `useSortable`을 사용하면 모든 카드가 항상 다시 렌더링되나요?

아닙니다. 이번 사례에서는 여러 카드가 같은 `SortableContext` 상태를 구독하는 구조와 카드 콘텐츠가 같은 컴포넌트에 있었던 점이 문제였습니다. 실제 영향 범위는 컴포넌트 구조와 구독 방식에 따라 달라집니다.

### `React.memo`는 Context 변경에 따른 렌더링을 막을 수 있나요?

컴포넌트가 직접 읽는 Context 값이 변경되면 `React.memo`를 사용해도 다시 렌더링될 수 있습니다. Context를 읽는 컴포넌트와 UI를 표시하는 컴포넌트를 분리하면 `memo`를 적용할 범위를 명확히 만들 수 있습니다.

### Render Props가 성능을 개선하는 기능인가요?

아닙니다. Render Props는 라이브러리가 계산한 드래그 관련 값을 함수 인자로 전달하는 API 패턴입니다. 이번 개선에서는 이 값을 드래그를 담당하는 컴포넌트에 한정해 전달하는 경계를 만드는 데 활용했습니다.

### React Portal은 렌더링 성능을 줄이기 위해 사용했나요?

아닙니다. 드래그 중인 카드가 스크롤 컨테이너와 `overflow`, `z-index`, 레이아웃의 영향을 덜 받도록 DOM 위치를 분리하기 위해 사용했습니다.
