## 2. 핵심 성과 및 기술적 도전

### 💬 WebSocket 실시간 채팅 성능 최적화 (Hybrid Communication Strategy)
- **배경**: 실시간 토론장 구현을 위해 초기 설계 시 모든 데이터(채팅 기록, 참여자 프로필 정보, 토론 상태 등)를 WebSocket 단일 채널로 실시간 송수신하려 했습니다. 그러나 다음과 같은 한계가 존재했습니다:
  1. **초기 로딩 시간 증가**: 과거의 대량 채팅 내역과 토론장 구성을 위한 데이터를 모두 동기식 웹소켓으로 로드해야 해서 초기 진입 딜레이가 발생했습니다.
  2. **프로필 이미지 렌더링 병목**: 메시지가 수신될 때마다 발신자의 프로필 이미지 정보도 같이 받아와 매번 다운로드 처리하며 UI가 늦게 갱신되었습니다.
- **해결 전략**:
  - **데이터 송수신 분리**: WebSocket은 오직 실시간 채팅 메시지 송수신 전송에만 집중하게 역할을 한정하고, 과거 대화 내역 및 토론 참여자 상세 정보는 HTTP `Rest API`로 이원화하여 분리 로드했습니다.
  - **이미지 캐싱**: 토론자 정보를 진입 시 Rest API로 일괄 조회한 뒤, `CachedNetworkImage` 라이브러리를 사용해 프로필 이미지를 메모리/디렉토리 캐시 처리하여 네트워크 재요청을 차단했습니다.
  - **단계적 초기화**: 과거 내역 조회 및 토론 상태 조회가 모두 완료될 때까지 로딩 스켈레톤을 띄우고, 모든 조회가 완료된 후 비동기로 WebSocket을 연결하여 데이터 정합성 꼬임을 방지했습니다.

### 🛠️ 유연한 API 모킹 환경 및 생명주기 관리
- **도전**: 백엔드 API 개발 완료 시점까지 프론트엔드 개발이 지연되는 병목 현상을 방지하고, 단기간에 신속하게 생명주기 안정성을 확보해야 했습니다.
- **해결 전략**:
  - 백엔드 연동 전에 Firebase Realtime Database를 래핑하여 자체 모의(Mock) 서버 레이어를 선구축하여 프론트엔드 독립 개발 생산성을 올렸습니다.
  - 생명주기 수동 관리에 따른 메모리 누수 실수를 원천 차단하기 위해 `Flutter Hooks`를 선제적으로 도입하여 자동으로 리소스 해제 및 컨트롤러 폐기를 처리하게 구성했습니다.
