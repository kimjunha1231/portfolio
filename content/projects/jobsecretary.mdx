## JobSecretary

**기간**: 2025.11 ~ 2025.12

JobSecretary는 채용 공고 확인부터 지원 일정 관리, 자기소개서 작성, 면접 준비까지의 과정을 하나의 흐름으로 연결한 AI 기반 취업 준비 관리 서비스입니다.

지원 현황은 칸반 보드로 관리하고, 자기소개서는 태그를 기준으로 저장·검색할 수 있도록 구성했습니다. 반복해서 작성해야 하는 자기소개서와 지원 정보를 한곳에 모아 필요한 순간 빠르게 꺼내 쓸 수 있도록 했습니다.

Gemini AI를 활용해 기존 자기소개서의 스타일과 데이터를 참고한 초안 작성, 문장 교정, 면접 질문 생성을 지원하도록 구현했습니다. AI가 만든 결과를 그대로 사용하는 방식이 아니라, 사용자가 최종 내용을 직접 검토하고 수정할 수 있는 흐름을 중심으로 설계했습니다.

1인 개발로 기획·설계·프론트엔드 개발·테스트·배포까지 진행하며 서비스의 전체 흐름을 직접 구성했습니다.

## 핵심 기능

**지원 현황 관리 :** 칸반 보드를 기반으로 지원 상태와 일정을 한눈에 확인할 수 있도록 구성했습니다. 지원 단계가 바뀌면 드래그 앤 드롭으로 카드를 이동할 수 있으며, 각 지원 건의 회사·직무·마감일과 현재 상태를 함께 확인할 수 있습니다.

**자기소개서 관리 :** 태그를 기준으로 작성한 내용을 저장하고 검색할 수 있도록 구성했습니다. 자기소개서 PDF 변환 기능을 제공하고 자기소개서 작성 시 태그로 이전 자기소개서를 확인할 수 있습니다.

**AI :** Gemini를 활용해 자기소개서 초안, 문장 교정, 면접 질문 생성을 보조하도록 구성했습니다. 반복적인 초안 작업은 AI가 돕고, 방향과 최종 내용은 사용자가 직접 판단하도록 역할을 나누었습니다.

## 개발 기록

### 1. 칸반 보드 렌더링 성능 최적화

**문제 상황** React Developer Tools로 드래그 동작을 측정한 결과, 카드 한 개를 이동할 때마다 271회의 커밋이 발생하고 모든 카드가 함께 리렌더링되는 현상을 확인했습니다.

**원인** 초기 `dnd-kit` 구현에서는 `KanbanCard` 내부에서 `useSortable`을 호출해 각 카드가 `SortableContext`의 드래그 상태를 직접 구독하고 있었습니다. `dnd-kit`의 `useSortable`이 `SortableContext`의 상태를 직접 구독하는 구조에서는, `React.memo`만으로 드래그 중 카드 컴포넌트의 리렌더링을 막을 수 없었습니다.

**해결 과정** `dnd-kit`에서 `@hello-pangea/dnd`로 라이브러리를 교체하고, `Draggable`의 Render Props로 `provided`와 `snapshot`을 카드 컴포넌트에 전달했습니다.

실제 카드에 드래그 상태를 연결하는 부분은 다음과 같습니다.

```tsx
<Draggable draggableId={id} index={index}>
  {(provided, snapshot) => (
    <KanbanCard
      provided={provided}
      isDragging={snapshot.isDragging}
    />
  )}
</Draggable>
```

`provided`는 카드와 드래그 기능을 연결하고, `snapshot.isDragging`은 드래그 상태에 따른 스타일 처리에 사용했습니다. `KanbanCard`와 `KanbanCardContent`를 분리한 뒤 카드 콘텐츠에 `React.memo`를 적용했으며, 드래그 중인 카드는 `React Portal`로 `document.body`에 렌더링했습니다.

**결과** React Developer Tools 측정 기준 전체 커밋 횟수는 271회에서 98회로, 비활성 컴포넌트의 리렌더링은 268회에서 0회로 감소했습니다. 라이브러리 교체만의 효과로 표현하기보다, 드래그 상태 전달 방식 변경·컴포넌트 분리·`memo`·Portal을 함께 적용한 결과로 기록했습니다.

### 2. React Hook Form과 Zustand 선택

**문제 상황** 자기소개서 작성 중 새로고침이나 뒤로가기로 내용이 유실되지 않으면서도, 입력할 때마다 발생하는 리렌더링과 스토리지 접근을 줄여야 했습니다.

**원인** 초기에는 제어 컴포넌트와 `Zustand(persist)`를 결합해 타이핑마다 `localStorage`에 저장하는 방식을 고려했습니다. 이후 `React Hook Form`을 함께 사용하거나 `Debounce`를 적용하는 대안도 검토했지만, Zustand가 상태를 읽고 다시 저장하는 과정에서 불필요한 업데이트가 남을 수 있었습니다.

**해결 및 결과** 최종적으로 `React Hook Form`과 `useFieldArray`를 사용해 비제어 컴포넌트 기반으로 문항별 폼을 관리하고, 저장은 사용자가 완료한 시점에 서버에 반영하도록 분리했습니다. 새로고침이나 뒤로가기 시 데이터 유실 문제는 `isDirty` 상태를 기준으로 이탈 방지 Dialog를 표시하는 UX 방식으로 해결했습니다. 성능 최적화와 작성 중 데이터 보호를 서로 다른 책임으로 나누어 관리할 수 있었습니다.

### 3. 기능 확장을 고려한 FSD 구조

**문제 상황** 초기에는 App, Features, Entities, Shared 계층만 사용했지만, 기능이 늘어나면서 페이지 조립 코드와 기능별 로직, 공통 UI의 경계가 흐려졌습니다.

**원인** Widgets 계층이 없으니 Features 안에 View 폴더를 만들어 페이지와 위젯 역할을 대신하게 되었고, 한 Feature에서 다른 Feature의 slice를 직접 참조하는 의존성이 생겼습니다. 공통 사이드바 역시 Shared에 두기에는 자체 로직과 책임이 커 적절한 위치를 정하기 어려웠습니다.

**해결 및 결과** App, Widgets, Features, Entities, Shared 계층으로 구조를 재정리했습니다. 사이드바와 페이지 단위 UI는 Widgets로, 사용자 상호작용은 Features로, 비즈니스 데이터 모델은 Entities로 분리해 기능 간 의존성과 공통 UI의 책임을 정리했습니다. 이후 이력서·면접 복기·포트폴리오 관리 기능을 확장할 수 있는 기반도 마련했습니다.

### 4. 웹 성능·접근성·품질 관리

**문제 상황** PDF 변환처럼 특정 화면에서만 사용하는 기능이 초기 번들에 포함될 수 있었고, 접근성 누락과 1인 개발에서 발견하기 어려운 오류도 관리할 필요가 있었습니다.

**원인** PDF 생성 라이브러리를 모든 페이지에서 미리 불러오면 초기 로딩 부담이 커지고, 수동 확인만으로는 키보드·스크린 리더 사용성이나 예외 상황을 충분히 검증하기 어려웠습니다.

**해결 및 결과** PDF 다운로드 버튼을 클릭하는 시점에 `@react-pdf/renderer`를 Dynamic Import하고, Lazy Loading과 Skeleton UI를 적용해 초기 로딩 부담을 줄였습니다. `aria-label`, `label`, `sr-only`를 적용해 접근성도 보완했으며, Jest와 Playwright로 테스트 환경을 구성하고 Sentry로 배포 후 오류를 모니터링했습니다. Lighthouse 접근성 점수는 80점대에서 95점대로 개선했습니다.
