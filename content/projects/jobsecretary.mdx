## 프로젝트 개요

- **구분**: 개인 프로젝트 (1인 개발)
- **기간**: 2025.11 ~ 2025.12
- **담당 범위**: 기획·설계·프론트엔드 개발·테스트·배포
- **기여도**: 100%
- **기술 스택**:
  - 프론트엔드: Next.js, TypeScript
  - 폼 상태 관리: React Hook Form, Zod
  - 서버 상태 관리: TanStack Query
  - 전역 상태 관리: Zustand
  - 스타일링: Tailwind CSS, shadcn/ui
  - 테스트·모니터링: Jest, Playwright, Sentry
  - 백엔드·데이터: Next.js API Routes, Supabase
  - AI·배포: Google Gemini API, Vercel
- **주요 성과**:
  - React Developer Tools Profiler의 동일한 드래그 시나리오에서 전체 커밋 **271회 → 98회** (**63.8% 감소**), 비활성 컴포넌트 리렌더링 **268회 → 0회** (**100% 감소**)
  - Lighthouse 접근성 점수 **80점대 → 95점대** (**15점 이상 향상**)

JobSecretary는 채용 공고와 지원 현황, 자기소개서, 면접 준비까지의 과정을 하나로 연결한 AI 기반 취업 준비 관리 서비스입니다. 칸반 보드와 태그 검색으로 지원 정보와 자기소개서를 관리하고, Gemini로 초안·문장 교정·면접 질문 생성을 지원합니다.

## 담당 기능

- **지원 현황 관리**: 칸반 보드에서 지원 단계와 일정을 확인하고, 드래그 앤 드롭으로 지원 상태를 변경할 수 있도록 구성했습니다.
- **자기소개서 관리**: 자기소개서를 태그로 저장·검색하고, PDF 변환과 이전 문서 참고 기능을 제공했습니다.
- **AI 보조**: Gemini를 활용해 자기소개서 초안·문장 교정·면접 질문 생성을 지원하되, 최종 판단은 사용자가 직접 하도록 구성했습니다.

## 개발 기록

### 1. 칸반 보드 렌더링 성능 최적화

- **문제 상황**: React Developer Tools Profiler에서 동일한 칸반 카드 목록을 드래그해 다른 위치에 놓는 동작을 측정했을 때, 한 번의 드래그 과정에서 271회의 커밋이 발생하고 비활성 카드도 268회 렌더링되었습니다.
- **원인**: 초기 `dnd-kit` 구현에서 `useSortable`을 사용하는 카드가 `SortableContext`의 드래그 상태를 직접 구독했고, 드래그 상태와 카드 콘텐츠가 같은 컴포넌트에 섞여 있었습니다.
- **해결 과정**: `@hello-pangea/dnd`로 교체하고 `Draggable`의 Render Props로 드래그 연결 정보와 상태를 전달했습니다. 카드 콘텐츠를 별도 컴포넌트로 분리해 `React.memo`를 적용했으며, Portal은 성능 개선이 아니라 드래그 중 `overflow`·`z-index`·레이아웃 간섭을 줄이는 용도로 사용했습니다.

드래그 상태를 카드에 연결하는 방식은 다음과 같습니다.

```tsx
<Draggable draggableId={id} index={index}>
  {(provided, snapshot) => (
    <KanbanCard
      provided={provided}
      isDragging={snapshot.isDragging}
    />
  )}
</Draggable>
```

- **결과**: 전체 커밋 횟수는 **271회 → 98회** (**63.8% 감소**), 비활성 컴포넌트 리렌더링은 **268회 → 0회** (**100% 감소**)로 대폭 감소했습니다.

### 2. 자기소개서 편집 상태 관리

- **문제 상황**: 긴 자기소개서를 여러 문항으로 작성하는 동안 입력 상태와 서버 저장 상태를 분리하면서도, 작성 중 이탈로 인한 데이터 손실을 안내해야 했습니다.
- **해결 과정**: `useForm`과 `useFieldArray`를 사용해 문항별 입력을 비제어 방식으로 관리하고, 저장 버튼을 눌렀을 때만 mutation으로 서버에 반영했습니다. `isDirty` 상태를 기준으로 새로고침·뒤로가기·목록 이동 시 이탈 경고를 표시했습니다.
- **결과**: 입력 상태와 저장 책임을 분리하고, 작성 중인 문서가 저장되기 전 이탈할 때 사용자에게 명확한 경고를 제공했습니다.

### 3. 기능 재사용과 확장을 고려한 FSD 구조

- **문제 상황**: 기능이 늘어나면서 페이지를 구성하는 UI와 사용자 동작을 처리하는 코드가 같은 계층에 섞이기 시작했습니다. 특히 자기소개서 작성·수정 화면에서 공통으로 사용하는 AI 문장 교정·초안 생성과 태그 기반 참고 문서 검색 기능을 여러 화면에서 재사용하기 어려웠습니다.
- **원인**: 초기 구조에는 Widgets 계층이 없어 Features 내부에서 화면 조립과 기능 로직을 함께 처리했습니다. 그 결과 Feature 내부 모듈을 다른 Feature가 직접 참조하는 의존성이 생겼고, 사이드바처럼 자체 상태와 동작을 가진 공통 UI를 어느 계층에 둘지도 모호했습니다.
- **해결 과정**: 단순히 폴더를 FSD 규칙에 맞추는 데 그치지 않고, 실제 코드의 재사용 범위와 계층별 책임을 기준으로 구조를 다시 설계했습니다. App, Widgets, Features, Entities, Shared로 계층을 재구성하고, 페이지 조립·사이드바·보드 UI는 Widgets에, 여러 화면에서 사용하는 AI·참고 문서 검색 기능은 Features에, 공통 데이터 모델은 Entities에 배치했습니다.
- **결과**: 작성·수정 화면에서 AI 문장 교정·초안 생성·태그 검색 기능을 중복 구현하지 않고 재사용할 수 있게 되었습니다. 계층별 책임과 의존성 방향도 명확해져 기능을 추가하거나 수정할 때 변경 범위를 쉽게 파악할 수 있었습니다.

### 4. PDF 기능 분리를 통한 초기 번들 최적화

- **문제 상황**: PDF 변환은 다운로드 시점에만 필요한 기능인데, 관련 라이브러리가 초기 번들에 포함되어 PDF를 사용하지 않는 사용자도 해당 코드를 내려받고 있었습니다.
- **원인**: PDF 라이브러리를 페이지 진입 시 미리 불러오는 구조로 인해 초기 번들 크기와 로딩 부담이 커질 수 있었습니다.
- **해결 과정**: PDF 다운로드 시점에 `@react-pdf/renderer`를 Dynamic Import하고, Lazy Loading과 Skeleton UI를 적용했습니다.
- **결과**: PDF 기능을 실제로 사용하는 시점에만 관련 코드를 불러오도록 초기 로딩 부담을 줄였습니다.

### 5. 접근성·테스트·배포 품질 관리

- **문제 상황**: 키보드·스크린 리더 사용성을 보완하면서, 1인 개발 환경에서 발생할 수 있는 기능 오류와 배포 후 문제도 지속적으로 확인해야 했습니다.
- **원인**: 수동 확인만으로는 접근성 누락과 예외 상황을 일관되게 검증하기 어려웠습니다.
- **해결 과정**: `aria-label`, `label`, `sr-only`를 적용해 접근성을 보완하고, Jest와 Playwright로 테스트 환경을 구성했습니다. Sentry를 연동해 배포 후 오류도 모니터링했습니다.
- **결과**: Lighthouse 접근성 점수를 **80점대 → 95점대** (**15점 이상 향상**)로 개선하고, 기능 오류를 테스트와 배포 후 모니터링으로 확인할 수 있게 되었습니다.
